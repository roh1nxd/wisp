## Overview
The Voting dApp is a decentralized application built on the Stellar network using Soroban smart contracts. It allows an administrator to initialize a poll with a specific set of options. Users can then connect their Freighter wallets to cast a single vote for their preferred option. The contract ensures integrity by preventing double-voting and requiring cryptographic authorization for every vote cast.

## Core Entities
- **Poll**: The global state containing the poll's title and the list of available options.
- **Option**: A structure representing a candidate or choice, consisting of a name (string) and a vote count (integer).
- **Voter Record**: A mapping of user addresses to a boolean or option index to ensure each address votes exactly once.
- **Admin**: The address responsible for initializing the contract state.

## User Roles
- **Administrator**: The entity that deploys and initializes the contract with the poll question and options.
- **Voter**: Any user with a Stellar wallet (Freighter) who interacts with the contract to submit a vote.

## Features (v0 vs v1)
### v0 (Current Draft)
- **Single Poll Support**: The contract manages one active poll.
- **Fixed Options**: Options are defined at the time of initialization.
- **One-Vote-Per-Address**: Strict enforcement of unique voting.
- **Real-time Results**: Public functions to fetch current vote tallies.

### v1 (Target MVP)
- **Expiration Timestamps**: Polls that automatically close after a certain ledger height or time.
- **Multi-Poll Factory**: A factory contract to deploy multiple independent voting instances.
- **Weighted Voting**: Integration with Soroban tokens (e.g., voting power based on token balance).
- **Admin Dashboard**: Frontend interface for poll creation and management.

## Architecture
- **Smart Contract (Soroban/Rust)**:
    - **Storage**: Uses `Persistent` storage for the poll configuration and `Persistent` (or `Instance`) storage for the vote counts.
    - **Auth**: Utilizes `address.require_auth()` to verify the identity of the voter.
    - **Data Types**: `Symbol` for option names, `Map` or `Vec` for tracking results.
- **Frontend (HTML5/JavaScript)**:
    - **Stellar SDK**: Used to construct `invokeContractFunction` operations and handle XDR encoding/decoding.
    - **Freighter API**: Used for secure transaction signing without exposing private keys.
    - **State Management**: Local UI state to track transaction status (Pending, Success, Error).

## Roadmap (v0 to v1 steps)
1.  **Contract Development**: Implement `initialize(admin, options)` and `vote(voter, option_index)`.
2.  **Security Audit**: Ensure `require_auth()` is correctly placed to prevent "voting on behalf of others."
3.  **Frontend Integration**: Build the JS layer to fetch contract data (results) and trigger the `vote` function.
4.  **Deployment**: Deploy to Stellar Testnet and initialize the contract ID.
5.  **V1 Enhancements**: Add time-locking logic to the contract and a "Create Poll" UI for the admin.

## Open Questions
- **Storage Costs**: How will the contract handle storage rent if thousands of unique voters participate? (Likely requires `extend_ttl`).
- **Privacy**: Should the votes be public or is there a requirement for encrypted/blinded voting? (v1 assumes public transparency).
- **Option Limits**: What is the maximum number of options a single poll can support before hitting Soroban resource limits? (Recommended limit: 10-20 for v1).
